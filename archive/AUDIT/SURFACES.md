# Surface Inventory — Phase 8A

Every runtime surface Recall ships today, with its entry
point, owner file, and live status. Generated by walking
`recall.py`'s subcommand dispatch + each frontend's `App` /
build target.

> Status legend: **LIVE** = primary code path · **LEGACY**
> = behind env-var escape hatch or pre-v3 compat · **ARCHIVE**
> = snapshot only · **REMOVE** = dead, safe to delete after
> grace period.

## Search overlay (extension) vs launcher search

The directive lists both as separate surfaces. Today:

- **Launcher search bar** ([`MinimalSearchBar`](../../app/ui/launcher_v3/minimal.py)) — emits `query_changed` + `searchChanged` (Phase 7E.1 frozen contract). Wired to `LiveLauncher._on_query_changed` → `_request_search` signal. The legacy launcher consumes this fully; **the v3 launcher's search results panel is unwired** (placeholder — typing emits the signal but no results render). LIVE-but-incomplete.
- **Extension search overlay** — Cmd/Ctrl+K opens
  `SearchOverlay`; in-memory filter across the popup's
  loaded state. LIVE.

## What this audit confirms

- The launcher has **one live surface** (v3) + **one
  legacy escape hatch** (`RECALL_LAUNCHER=legacy`). Both
  are real code paths. No third launcher exists.
- The extension has **one live composition** (`v2/`) + a
  pre-7A component tree that no longer renders. The
  pre-7A components could be removed once the legacy
  smoke tests stop touching them.
- The daemon's `/v1/*` routes are mostly consumed; a few
  (`thread_forget`, `contexts/recent`, `sessions/recent`,
  `events/recent`) have no live consumer — see
  [`DEAD_CODE.md`](DEAD_CODE.md).
- Demo mode + onboarding + alpha tooling are all live and
  thinly coupled — none of them gate the main launcher
  path.
- The Mac build is **not** a live surface yet — see
  [`docs/release/MAC_BUILD_STATUS.md`](../../docs/release/MAC_BUILD_STATUS.md).
